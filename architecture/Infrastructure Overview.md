# Infrastructure Overview

## Deployment topology

```
                    ┌─────────────────────┐
                    │   Cloudflare DNS    │
                    │   *.kodama.page     │
                    └─────────┬───────────┘
                              │
              ┌───────────────┼───────────────┐
              │               │               │
    ┌─────────▼──────┐ ┌─────▼──────┐ ┌──────▼───────┐
    │ Cloudflare     │ │ Cloudflare │ │  Supabase    │
    │ Pages          │ │ CDN        │ │  Edge Func   │
    │                │ │            │ │              │
    │ Product SPAs   │ │ Static     │ │ Platform API │
    │ Marketing site │ │ assets     │ │ PostgreSQL   │
    └────────────────┘ │            │ │ Storage      │
                       └────────────┘ └──────────────┘
```

## Compute

### Static frontends

All product SPAs and the marketing site are static builds deployed to **Cloudflare Pages**:

- Build output: `dist/` directory
- Build command: `npm run build`
- No server-side rendering (except `brand.kodama.page` which uses Next.js)
- HTTPS is required for Web Crypto API (Note encryption)

### Platform API

The Platform API runs as a **Supabase Edge Function** (`kodama-api`):

- Runtime: Deno
- Framework: Hono 4
- Deployed to: `api.kodama.page`
- Local development: Deno server on port 8000

The API can also run as a standalone Deno server for local development without Supabase.

## Database

**Supabase PostgreSQL 15** — a single shared database for all product types.

| Component | Details |
|---|---|
| Provider | Supabase (managed PostgreSQL) |
| ORM | Drizzle ORM |
| Migrations | SQL files in `supabase/migrations/` |
| Local dev | `supabase start` (Docker) |
| Connection | Service role key (API only, never exposed to frontends) |

Key tables:

| Table | Purpose |
|---|---|
| `kodama_resources` | All Place instances across product types |
| `kodama_resource_versions` | Version history for versioned products |
| `kodama_drop_messages` | Drop inbox messages |
| `kodama_sessions` | Server-side session tokens |
| `kodama_drop_message_fingerprints` | Message deduplication |
| `api_rate_limits` | IP-based rate limiting buckets |
| `kodama_reserved_slugs` | Globally reserved slug list |

## Object storage

**Supabase Storage** for file attachments:

- Bucket: `page-attachments` (Note)
- Access: Signed URLs generated by the Platform API
- Future: Direct-to-storage upload for large ciphertext (Security Protocol)

## CDN

**Cloudflare CDN** at `cdn.kodama.page`:

- Shared icons, images, and static assets
- Referenced by all product frontends and the marketing site
- Cache headers set at upload time

## Network security

| Control | Implementation |
|---|---|
| CORS | Restricted to `*.kodama.page` + local dev origins |
| Rate limiting | IP-hash buckets in `api_rate_limits` table |
| HTTPS | Enforced on all domains via Cloudflare |
| API auth | Service role key (server-side only) |
| Session tokens | Short-lived, product-scoped, stored in `kodama_sessions` |

## Environment separation

| Environment | API | Database | Frontends |
|---|---|---|---|
| **Production** | `api.kodama.page` | Supabase production project | `*.kodama.page` |
| **Local dev** | `localhost:8000` | `supabase start` (Docker) | `localhost:5173` (Vite) |
| **Preview** | Staging API (planned) | Supabase branch (planned) | Cloudflare Pages preview |

Local development requires two terminals:

```bash
# Terminal 1: API + database
npm run db:start && npm run dev:api

# Terminal 2: Frontend
npm run dev
```

## Monitoring and observability

See [Monitoring](../operations/Monitoring.md) for operational details.

At the infrastructure level:

- **Cloudflare Analytics** — traffic, cache hit rates, security events
- **Supabase Dashboard** — database metrics, Edge Function logs, storage usage
- **Application logs** — structured JSON from Hono middleware

## Scalability considerations

The current architecture is designed for simplicity and low operational overhead:

- Static frontends scale infinitely via CDN
- Edge Functions scale with Supabase's infrastructure
- PostgreSQL handles the expected load for Place-based workloads
- Rate limiting prevents abuse without complex infrastructure

If scale demands increase:

1. Add read replicas for PostgreSQL
2. Move object storage to a dedicated provider (S3/R2)
3. Add a caching layer (Cloudflare KV) for hot reads
4. Split Edge Functions by product type if cold-start latency becomes an issue
